# SKILL: Database Schema Design for Anti-Fraud Analytics

## Overview

The database stores 5 levels of data about Google Ads campaigns and their outcomes (bans). The schema must support:
1. High-volume inserts from Chrome Extensions (100+ profiles sending data)
2. Analytical queries (correlations, aggregations, time-series)
3. Graph traversal (which accounts share consumables — for ban chain prediction)
4. Raw data preservation (always keep original payloads)

## Schema Design Principles

### Primary Keys
- Use `UUID` generated by Postgres (`gen_random_uuid()`)
- Never use auto-increment — it leaks information about record count and ordering

### Timestamps
- Every table gets `created_at TIMESTAMPTZ DEFAULT NOW()` and `updated_at TIMESTAMPTZ DEFAULT NOW()`
- Create a reusable trigger function for auto-updating `updated_at`:

```sql
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

### JSONB Usage
- Use JSONB for: target_geos, raw_payload, ad_texts, extension metadata
- Always define TypeScript interface for expected JSONB shape
- Add GIN indexes on JSONB columns that will be queried: `CREATE INDEX idx_target_geos ON campaigns USING GIN (target_geos)`

### Enums
- Define as Postgres ENUMs in migrations
- Keep TypeScript enums in `packages/shared` in sync
- When adding new enum values, use `ALTER TYPE ... ADD VALUE` (never recreate)

### Indexes Strategy
- All foreign keys get indexes automatically with Knex `.references()`
- Add explicit indexes on: `domain`, `google_account_id`, `banned_at`, `offer_vertical`, `campaign_type`
- Partial indexes for common filters: `CREATE INDEX idx_active_campaigns ON campaigns (id) WHERE banned_at IS NULL`
- Composite indexes for common query patterns: `(offer_vertical, banned_at)` for "ban rate by vertical"

## Entity Relationship Summary

```
domains
  └── accounts (many accounts can use same domain)
        ├── campaigns (one account has many campaigns)
        │     └── ban_events (one campaign can have ban history)
        ├── account_consumables → proxies
        │                       → antidetect_profiles
        │                       → payment_methods
        └── raw_payloads (all intercepted data)

predictions (references campaigns or accounts)
ai_leaderboard (aggregated model scores)
```

## Performance Considerations

- `raw_payloads` will be the largest table — consider partitioning by month if it grows past 1M rows
- Use `COPY` or batch inserts for bulk data import
- Connection pooling via `pg` pool (max 20 connections for start)
- Avoid N+1 queries — use JOINs or batch lookups in repositories
