# CLAUDE.md — Project Instructions for Claude Code

## Project Overview

**CTS Anti-Fraud Analytics** — internal platform for Google Ads campaign monitoring, ban prediction, and anti-fraud intelligence. Built for a media buying team running Google Ads through anti-detect browsers.

**Architecture:** Chrome Extension (in anti-detect browser) intercepts Google Ads XHR/fetch responses → sends to Fastify backend → stores in PostgreSQL → dashboard visualizes. NO Google Ads API used — intentional design to avoid account linking.

## Tech Stack

- **Monorepo:** npm workspaces
- **Language:** TypeScript strict mode everywhere
- **Backend:** Fastify + Knex.js (migrations) + pg (raw queries)
- **Frontend:** React 18 + Tailwind CSS + Vite
- **Database:** PostgreSQL 15+
- **Extension:** Chrome Manifest V3, vanilla TypeScript
- **Runtime:** Node.js 20+

## Code Conventions

### Naming
- **Variables/functions:** camelCase
- **Types/Interfaces:** PascalCase
- **Constants/Enums:** SCREAMING_SNAKE_CASE
- **Files:** kebab-case (e.g., `ban-log.service.ts`)
- **Migrations:** `YYYYMMDD_NNN_description.ts` (e.g., `20260216_001_create_domains.ts`)

### Architecture (server)
```
packages/server/src/
├── routes/          # Fastify route definitions (thin — just schema + handler call)
├── handlers/        # Request/response handling, validation
├── services/        # Business logic
├── repositories/    # Database queries (raw SQL via pg)
├── migrations/      # Knex migration files
├── plugins/         # Fastify plugins (auth, rate-limit, etc.)
├── config/          # Environment config
└── index.ts         # Entry point
```

### Architecture (extension)
```
packages/extension/src/
├── background/      # Service worker
├── content/         # Content scripts (injected into ads.google.com)
├── popup/           # Extension popup UI
├── interceptors/    # Fetch/XHR interception logic
├── collectors/      # Data extraction and normalization
├── transport/       # Batching, retry, queue management
└── types/           # Extension-specific types
```

### TypeScript Rules
- `strict: true` always
- No `any` — use `unknown` and narrow
- Prefer `interface` over `type` for object shapes
- Export types from `packages/shared` — import everywhere else
- Use branded types for IDs: `type AccountId = string & { __brand: 'AccountId' }`

### Database Rules
- UUIDs for all primary keys (generated by Postgres: `gen_random_uuid()`)
- Every table has `created_at` and `updated_at` (with trigger for auto-update)
- Use JSONB for flexible structures, but always define the expected shape in TypeScript
- Indexes on: foreign keys, columns used in WHERE/ORDER BY, enum columns used for filtering
- `raw_payload JSONB` on tables receiving extension data — always preserve original data

### API Rules
- All endpoints under `/api/v1/`
- Use Fastify JSON Schema validation on all routes
- Consistent error format: `{ error: string, code: string, details?: unknown }`
- Rate limiting per-route where needed
- API key auth via `X-API-Key` header

### Extension Rules
- Zero external dependencies — vanilla TS only
- Minimal permissions — only `ads.google.com` host
- Buffer intercepted data, send in batches (every 30s or on tab close)
- Queue unsent data in `chrome.storage.local`
- Retry with exponential backoff

## Important Context

- This is an INTERNAL tool — no public auth, no signup, no multi-tenancy
- Extension runs inside anti-detect browsers (Chromium-based) — must work with Manifest V3
- Each Google Ads account is in a separate browser profile — extensions send data independently
- Google must NOT detect that accounts are linked — zero shared identifiers
- The team has experience with AdsPower, Dolphin Anty, Octo Browser
- Russian-speaking team — comments in code can be English, UI will be Russian later
